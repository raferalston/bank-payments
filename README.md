# Запуск проекта и использование
- Дисклеймер: MVP (не продакш) среда. Пример использования технологий для решения поставленной задачи.

### Требования

- **Python 3.13+**
- **Docker и Docker Compose** (для PostgreSQL и Redis)
- **uv** — менеджер пакетов ([установка](https://docs.astral.sh/uv/getting-started/installation/))

### 1. Клонирование и настройка окружения

```bash
# Установка зависимостей
make install
# или: uv sync --all-groups
```

### 2. Переменные окружения

Создайте файл `.env` в корне проекта (или скопируйте из примера):

```env
DEBUG=True
DATABASE_URL=postgres://it-guru:it-guru@localhost:5433/it-guru
REDIS_URL=redis://localhost:6379/0
BANK_API_URL=http://localhost:8001
BANK_REQUEST_TIMEOUT=10.0
BANK_MAX_RETRIES=3
```

- `DATABASE_URL` — строка подключения к PostgreSQL (порт 5433 соответствует маппингу в docker-compose).
- `TEST_DATABASE_URL` — БД для тестов; если не задана, используется БД с именем `{имя_из_DATABASE_URL}_test`.
- `BANK_API_URL` — URL мока банка (по умолчанию `http://localhost:8001`).

### 3. Запуск инфраструктуры (БД и Redis)

```bash
# Запустить только PostgreSQL и Redis
make db
# или: docker compose up -d db redis
```

Проверьте, что контейнеры запущены: `docker compose ps`.

### 4. Миграции базы данных

```bash
# Применение миграций
make migrate
```

### 5. Запуск приложения

Нужны **три процесса** (в разных терминалах):

**Терминал 1 — основное API:**

```bash
make app
# или: uv run uvicorn src.main:app --reload --port 8000
```

API доступно по адресу: **http://localhost:8000**

**Терминал 2 — мок банка (для эквайринга):**

```bash
make bank
# или: uv run uvicorn bank_mock.main:app --reload --port 8001
```

Мок банка: **http://localhost:8001**

**Терминал 3 (опционально) — Celery worker и Flower:**

```bash
# Воркер фоновых задач
make celery

# В другом терминале — мониторинг Celery
make flower
# Flower: http://localhost:5555
```

### 6. Полный запуск через Docker (альтернатива)

Поднять только БД и Redis, приложение — локально (как выше):

```bash
make up    # docker compose up -d
make db    # если нужны только db + redis
```

Остановка:

```bash
make down  # docker compose down
```

## Использование API

При `DEBUG=True` доступна интерактивная документация:

- **Swagger UI:** http://localhost:8000/docs
- **ReDoc:** http://localhost:8000/redoc

### Заказы (Orders)

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/orders` | Список всех заказов |
| GET | `/orders/{order_id}` | Детали заказа по ID |

У каждого заказа есть поля: `id`, `amount` (сумма), `payment_status` (`not_paid` / `partially_paid` / `paid`), `created_at`, `updated_at`.

### Платежи (Payments)

| Метод | URL | Описание |
|-------|-----|----------|
| POST | `/payments` | Создать платёж (депозит) |
| GET | `/payments/by-order/{order_id}` | Список платежей по заказу |
| GET | `/payments/remaining-balance/{order_id}` | Остаток к оплате по заказу |
| POST | `/payments/{payment_id}/refund` | Возврат платежа |
| POST | `/payments/{payment_id}/confirm` | Подтвердить платёж эквайринга (после оплаты в банке) |

**Создание платежа** — тело запроса (JSON):

```json
{
  "order_id": 1,
  "amount": "100.50",
  "payment_type": "cash"
}
```

или для эквайринга:

```json
{
  "order_id": 1,
  "amount": "500.00",
  "payment_type": "acquiring"
}
```

- `payment_type`: `"cash"` (наличные) или `"acquiring"` (эквайринг через банк).
- Сумма всех платежей по заказу не может превышать сумму заказа.

**Статусы платежа** (`operation_status`): `pending`, `deposited`, `refunded`, `failed`.

Для эквайринга: после создания платёж в статусе `pending`; после оплаты на стороне банка нужно вызвать `POST /payments/{id}/confirm` для синхронизации статуса.

### Пример сценария

1. `GET /orders` — посмотреть заказы, запомнить `order_id` и `amount`.
2. `GET /payments/remaining-balance/{order_id}` — узнать остаток к оплате.
3. `POST /payments` с `order_id`, `amount`, `payment_type` — создать платёж (наличные или эквайринг).
4. Для эквайринга: симулировать оплату в моке банка, затем `POST /payments/{id}/confirm`.
5. `GET /orders/{order_id}` — проверить обновлённый `payment_status`.
6. При необходимости: `POST /payments/{id}/refund` — возврат.

## ТЗ

1. Спроектировать и реализовать сервис работы с платежами по заказу

- В сервисе находится список из одного или более заказов, работа с которыми и формирование новых не требуется. Каждый заказ содержит поля "сумма заказа" и "статус оплаты" (не оплачен, частично оплачен, оплачен).
- Для каждого неоплаченного заказа должна быть возможность оплатить заказ одним или несколькими платежами.
- Каждый платеж должен иметь тип (наличные, эквайринг).
- У модели платежа должны быть реализованы две операции: депозит и возврат, которые должны в том числе согласованно менять статус оплаты заказа.
- Названия полей и их типы — на усмотрение кандидата.
- Сумма всех платежей по заказу не должна превышать сумму всего заказа.
- Модели платежей разных типов не должны отличаться друг от друга, т.е. работать с платежами разных типов должно быть возможно не задумываясь об их конкретном типе там, где это не важно.


2. API банка

- Для платежа через банк нужно предложить архитектурное решение, позволяющее:
  - вызывать API банка для создания платежа;
  - хранить состояние этого платежа;
  - согласовывать его состояние в нашем приложении и на стороне банка.

Это API не нужно никак реализовывать, реализовать только запросы к нему, руководствуясь нижеуказанными правилами.

Состоит из следующих ресурсов:

 2.1. bank.api/acquiring_start

- Принимает: JSON с номером заказа, суммой заказа.
- Возвращает:
  - в случае успеха — уникальный ID платежа в банке;
  - в случае неудачи — строку с ошибкой.

2.2. bank.api/acquiring_check`

- Принимает: уникальный ID платежа в банке.
- Возвращает: ID платежа в банке, сумму, статус, дату и время оплаты, либо ошибку «платеж не найден».

Требования

- Написать код обращения к этому API с обработкой ошибок. Учесть, что API банка это внешняя система.
- Использовать этот код в операциях работы с платежами по заказу.
- Учитывать, что состояние платежей в банке может быть изменено на стороне банка неожиданно для нашего приложения.
- Конкретные названия полей произвольные, типы данных на усмотрение кандидата.


3. Состояния, операции и REST API

- Реализовать операции создания и возврата платежа по заказу с использованием сущностей заказа, платежа и других (при их наличии).
- Для этих операций реализовать RESTful-ручки с использованием любого Python веб-фреймворка на усмотрение кандидата.
- В коде ручек нужно:
  - проверять входные данные от пользователя;
  - обрабатывать ошибки внутренних операций с корректным формированием ответа.
- Учитывать, что некоторые операции кандидату будет необходимо вызывать не только из RESTful-ручек.



4. Хранение данных и прочее

- Данные по заказам, платежам и другим связанным сущностям — хранить в реляционной БД. К заданию необходимо приложить схему разработанной БД.
- Приветствуются: паттерны проектирования, архитектурные приемы, тесты, обоснованное использование AI, документация.
- Несущественные для решения задачи детали можно упрощать, существенность определяется самим кандидатом. Однако выбор, внимание к мелочам и творческий подход, позволяющий упростить реализацию задачи, будут предметом оценки.
